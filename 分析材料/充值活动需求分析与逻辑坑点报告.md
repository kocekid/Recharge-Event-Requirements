## 1. 需求内容总结

### 1.1 活动入口与默认页

- 活动入口：
  - 首页顶部 Banner
  - 房间悬浮窗
- 用户从任意入口进入活动页，默认进入 `Recharge Reward / 充值奖励` 页。
- 页面顶部：
  - 返回按钮
  - 标题：`Recharge Activity`
  - 右上角：`Rule`，跳转规则页
- 顶部固定展示“当月充值排行榜第一名用户信息”：
  - 展示头像、昵称
  - 若当月暂无榜单数据：显示占位头像，昵称为 `Waiting For You`
  - 不随 Daily / Weekly / Monthly 切换变化
  - 时间口径统一按 UTC+3 自然月

证据：`stay项目组.txt` L4-L9、L43-L57。

### 1.2 充值奖励页

- 一级 Tab：
  - `Recharge Reward`，默认选中
  - `Recharge Ranking`
- 二级周期 Tab：
  - `Daily`
  - `Weekly`
  - `Monthly`
  - 默认 `Daily`
- 切换周期后，以下数据联动刷新：
  - 我的累计充值
  - 进度条
  - 档位奖励
  - Top3 数据
- 我的累计充值卡片：
  - User ID
  - My Daily / Weekly / Monthly Recharge: `{amount}`
- 进度条：
  - 展示 Daily / Weekly / Monthly Progress
  - 右侧 Recharge 按钮，跳转充值页
  - 档位节点展示圆点、勾选状态、阈值数字
  - 达标后节点可点击，展示该节点奖励详情
- 档位奖励展示区：
  - 支持左右箭头切换档位奖励卡
  - 每张卡展示该档位对应奖励：icon + 名称/数量

证据：`stay项目组.txt` L61-L91。

### 1.3 档位奖励逻辑

- 普通奖励：达标自动发放。
- 定制类奖励：需联系客服领取。
- 补充需求：档位奖励发放逻辑改为：用户在某个统计周期内（日/周/月），每达到一个档位门槛，就立即发放该档位奖励。
- 奖励发放完成后，系统发送对应系统通知给用户。

证据：`stay项目组.txt` L10-L18、L36-L39。

### 1.4 充值排名页

- 排名页一级 Tab 选中：`Recharge Ranking`
- 二级 Tab：
  - Weekly Ranking
  - Monthly Ranking
  - 默认 Weekly
- 倒计时：`Countdown: {dd}D {hh}H {mm}M {ss}S`
- 倒计时结束时触发刷新：切换到新周期并清空上周期数据。
- 排行奖励区：
  - Weekly Ranking Reward / Monthly Ranking Reward
  - 展示 TOP 1 奖励
  - 支持切换查看 Top2-3、Top4-8 奖励
- 榜单列表：
  - 展示 20 名用户
  - 每行展示：头像、昵称
  - UI 排序：前 3，中间 3 个一排，共 5 排，最下面 2 个
  - 置底固定“我的排名条”：我的排名、我的累计充值值、Recharge 按钮
  - 未上榜时我的排名显示 `20+`

证据：`充值页面-榜单.txt` L6-L34。

### 1.5 榜单历史页

- 周历史：列表展示近 3 周，每条展示周期日期范围 + Top3。
- 月历史：列表展示近 3 月，每条展示月份 + Top3。

证据：`充值页面-榜单历史.txt` L6-L12。

### 1.6 统计时间与充值口径

- 统一按 UTC+3 展示与统计。
- 日累计：每日 00:00:00 ～ 24:00:00。
- 周累计：每周一 00:00:00 ～ 周日 24:00:00。
- 月累计：每月 1 日 00:00:00 ～ 当月最后一日 24:00:00。
- 统计充值金额包含：官方应用内充值 + 币商充值。
- 代理账号限制：代理水晶充值不参与本活动所有奖励与排行榜。
- 规则页口径：以金币到账时间为准统计累计充值金额，仅统计有效到账金币。
- 水晶交易计入说明：通过币商交易到账的金币正常计入，包括币商卖给自己、卖给别人产生的金币到账。

证据：`stay项目组.txt` L24-L35、L110-L121。

### 1.7 Top3 达标规则

- Top3 定义：在当前统计周期内，最先达成“当前所查看档位”门槛的前三名用户。
- 排序优先级：
  - 达成时间早者排名靠前
  - 达成时间相同（同秒）则按财富等级排序
- 入榜人数不足 3：
  - 0 人：Top3 区域展示空状态
  - 1 / 2 人：展示已达成用户，其余位置占位
- 切换档位时，同步切换该档位对应的“最先达成前三名”。

证据：`stay项目组.txt` L94-L107。

### 1.8 奖励档位概览

从 `充值档位.pdf` 视觉识别可见：

- 日充值档位大致包含：50 / 100 / 150 / 300 美金，对应 500,000 / 1,000,000 / 1,500,000 / 3,000,000 金币门槛。
- 周充值档位大致包含：200 / 400 / 600 / 800 / 1000 美金，对应 2,000,000 / 4,000,000 / 6,000,000 / 8,000,000 / 10,000,000 金币门槛。
- 月充值档位大致包含：600 / 1000 / 1500 / 2000 / 2500 / 3000 美金，对应 6,000,000 / 10,000,000 / 15,000,000 / 20,000,000 / 25,000,000 / 30,000,000 金币门槛。
- 奖励类型包括：VIP、头像框、进场特效、气泡、资料卡装饰、金币奖励、标签、座驾、勋章、靓号、Banner、开机屏等。
- 周榜奖励覆盖 Top1、Top2-3、Top4-8。
- 月榜奖励覆盖 Top1、Top2-3、Top4-8。

注意：`充值档位.pdf` 的 PDF 文本抽取几乎没有表格内容，以上来自视觉识别，具体奖励名称和数量需要产品再次核对原始表格。

## 2. P0 / P1 逻辑冲突与坑点

### 暗坑 1：周期结束清空上周期数据 vs 次日 01:00 发排行奖励，结算快照可能丢失

涉及位置：
- `stay项目组.txt` L11-L13：统计周期结束后，次日 01:00 自动发放排行奖励。
- `充值页面-榜单.txt` L12-L15：倒计时结束时触发刷新，切换到新周期并清空上周期数据。
- `充值页面-榜单历史.txt` L6-L12：历史榜单展示近 3 周 / 3 月 Top3。

核心问题：
- 如果倒计时一结束就“清空上周期数据”，但排行奖励要到次日 01:00 才发，系统必须先冻结结算快照。
- 否则可能出现：榜单已清空、新周期开始，但发奖任务找不到旧周期排名数据。
- 历史榜单也依赖旧周期 Top3，如果只清空不归档，历史页没数据。

具体场景：
- 周日 24:00 UTC+3 倒计时结束，页面立刻切到新周并清空上周榜单。
- 周一 01:00 自动发周榜奖励。
- 如果后端没有 `period_snapshot`，Top1 / Top2-3 / Top4-8 无法稳定发奖，用户投诉时也无法追溯。

风险等级：P0

建议处理方向：
- 周期结束时先生成不可变结算快照：周期 ID、分区、榜单 Top20、奖励 Top1/Top2-3/Top4-8、用户充值值、排名、快照时间。
- 页面“清空”只能理解为前端切换新周期展示，不能删除或覆盖旧周期数据。
- 发奖、通知、历史榜单都必须读取同一份快照。

### 暗坑 2：金币到账口径 vs 充值金额口径 vs 币商/代理限制，规则互相打架

涉及位置：
- `stay项目组.txt` L28-L35：充值金额包含官方应用内充值 + 币商充值；代理水晶充值不参与；通过水晶交易产生的金币到账正常计入。
- `stay项目组.txt` L115-L121：以金币到账时间为准统计累计充值金额；通过币商交易到账的金币正常计入，包括币商卖给自己、卖给别人。

核心问题：
- 文档一边说“代理水晶充值不参与本活动所有奖励与排行榜”，一边说“币商卖给自己产生的金币到账正常计入”。
- 如果“币商”就是代理账号，卖给自己是否计入会直接冲突。
- “充值金额”又和“金币到账”混用，可能导致开发不知道按支付订单金额、金币到账数量、还是水晶交易流水统计。

具体场景：
- 代理账号 A 通过水晶给自己转出 1,000,000 金币。
- 规则 A：代理水晶充值不参与活动，所以 A 不该涨充值值。
- 规则 B：币商卖给自己产生的金币到账计入，所以 A 应该涨充值值。
- 同一笔数据到底进不进榜，文档没说清楚。

风险等级：P0

建议处理方向：
- 明确“参与用户主体”：普通用户可参与；代理/币商账号自身是否可参与必须单独定义。
- 建议口径：
  - 普通用户通过官方充值到账金币：计入。
  - 普通用户通过币商购买金币到账：计入。
  - 代理/币商账号作为卖方产生的水晶/金币变动：不计入。
  - 代理/币商账号给自己刷流水：不计入，除非产品明确允许。
- 后端按到账金币事件统计，但必须带来源类型、卖方/买方账号类型、是否有效、是否撤销。

---

### 暗坑 3：新增英语区/阿语区分区，但榜单、Top1、奖励、历史页都没定义分区口径

涉及位置：
- `stay项目组.txt` L40：分区逻辑改动：需要分英语区与阿语区。
- `stay项目组.txt` L47-L56：顶部固定展示当月 Top1。
- `stay项目组.txt` L94-L107：Top3 按当前周期、当前档位达成前三展示。
- `充值页面-榜单.txt` L6-L34：Weekly / Monthly Ranking 页面结构。
- `充值页面-榜单历史.txt` L6-L12：近 3 周 / 3 月历史 Top3。

核心问题：
- 只写了“需要分英语区与阿语区”，但没有定义：
  - 页面上是否有分区 Tab？
  - 用户默认看哪个区？
  - 首页当月 Top1 是全站 Top1，还是当前用户所在区 Top1？
  - Top3 达标榜是否按区分开？
  - Weekly / Monthly Ranking 是否按区发两套奖励？
  - 历史榜单是否按区保存？
  - 用户换区、语言切换、地区信息为空时怎么算？

具体场景：
- 英语区用户充值 3000 美金，阿语区用户充值 2500 美金。
- 如果首页 Top1 展示全站，阿语区用户可能看到英语区 Top1。
- 如果榜单奖励按区发，则两个区都应有 Top1；如果全站发，则分区只是展示筛选。
- 文档没定义，会导致前端、后端、运营发奖三套理解。

风险等级：P0

建议处理方向：
- 定义活动榜单维度：`period_type + period_id + region + rank_type`。
- 明确分区奖励是否独立发放。建议：英语区、阿语区各自独立榜单、独立 Top3、独立排行奖励、独立历史。
- 首页 Top1 应明确为“当前用户所在分区的当月 Top1”，或明确写“全站当月 Top1”。
- 用户分区来源要固定：注册地区 / 当前语区 / 运营配置，不能随语言切换实时变。

### 暗坑 4：档位奖励“立即发放”与“普通自动 / 定制联系客服”没有拆开，容易误发定制奖励

涉及位置：
- `stay项目组.txt` L14-L16：普通奖励达标自动发放；定制类奖励需联系客服领取。
- `stay项目组.txt` L36-L39：每达到一个档位门槛，就立即发放该档位奖励。
- `stay项目组.txt` L122-L123：Pretty ID、定制类奖励需联系客服领取。

核心问题：
- “立即发放该档位奖励”容易被开发理解为该档位下所有奖励都自动发。
- 但 Pretty ID、定制类奖励需要联系客服领取。
- 如果档位里既有自动道具，又有定制权益，必须拆成“自动发放奖励”和“人工领取权益”。

具体场景：
- 用户本月达到 3000 美金档，档位包含 VIP、头像框、Pretty ID 或靓号。
- 系统立即自动发 VIP 和头像框没问题。
- 但 Pretty ID / 靓号如果需要客服确认库存、选择号码、人工绑定，就不能自动发。

风险等级：P0

建议处理方向：
- 奖励表必须给每个 reward_item 增加字段：`发放方式=自动/联系客服/运营手动`。
- 立即发放只适用于自动奖励。
- 定制类奖励达标后生成“待领取资格”，用户联系客服后由后台核销。
- 定制权益必须定义领取截止时间、客服核验字段、核销状态、过期规则。

### 暗坑 5：Top3 达标榜排序规则不完整，同秒且财富等级相同会撞车

涉及位置：
- `stay项目组.txt` L94-L107：Top3 按最先达成档位门槛排序；同秒按财富等级排序。

核心问题：
- 同秒达成且财富等级相同，没有最终排序规则。
- 如果系统时间精度只到秒，高峰期同秒达标很常见。
- 没有稳定排序会导致前端展示、发奖任务、历史记录三边不一致。

具体场景：
- A、B、C、D 四个用户同一秒达到 1000 美金档，财富等级都为 45。
- 前端按接口返回顺序展示 A/B/C，发奖任务按数据库默认顺序发给 B/C/D。
- 用户截图对不上，投诉无法解释。

风险等级：P1

建议处理方向：
- Top3 最终排序建议：`达成时间毫秒 asc > 财富等级 desc > 达成时累计充值金额 desc > user_id asc`。
- 达成时间必须落库，不要实时计算。
- 每个档位、每个周期、每个分区只记录一次首次达成时间。

### 暗坑 6：Recharge Ranking 排行榜没有定义排序规则、并列规则、金额展示口径

涉及位置：
- `充值页面-榜单.txt` L25-L34：榜单列表展示 20 名用户；每行展示头像、昵称；我的排名条展示我的累计充值值。

核心问题：
- 周榜/月榜大概率应该按累计充值金额排序，但文档没有写。
- 金额相同怎么排？按到账时间、最后充值时间、财富等级，还是 UID？没定义。
- 每行字段只写头像、昵称，但 UI/我的排名又展示累计充值值，榜单行是否也展示金额不清晰。

具体场景：
- A 和 B 周累计都 10,000,000 金币。
- 如果 A 先达到，B 后达到，周榜 Top1 应该是谁？
- 排行奖励涉及 Top1 / Top2-3 / Top4-8，不定义并列规则就是直接埋雷。

风险等级：P0

建议处理方向：
- 周/月排名排序建议：`累计有效到账金币 desc > 达到当前累计值的时间 asc > 财富等级 desc > user_id asc`。
- 榜单行建议明确展示：排名、头像、昵称、累计充值值、分区、是否本人。
- 榜单奖励发放读取周期结束快照排名，而不是页面实时排序。

### 暗坑 7：统计时间写 24:00:00，数据库实现容易多算/少算边界

涉及位置：
- `stay项目组.txt` L24-L27：日/周/月累计时间写 00:00:00 ～ 24:00:00。
- `stay项目组.txt` L110-L114：规则页也写 00:00–24:00。

核心问题：
- `24:00:00` 不是多数数据库/语言中的常规时间点。
- 如果开发用 `<= 24:00:00` 或 `<= 当天 23:59:59`，会在毫秒/微秒数据上出错。
- 周日 24:00 与周一 00:00 是同一时刻，必须用左闭右开区间。

具体场景：
- 用户在 2026-05-09 00:00:00.500 UTC+3 到账。
- 如果日统计写 `<= 2026-05-09 00:00:00`，这笔不算。
- 如果写 `<= 2026-05-09 23:59:59` 又变成多算一天。

风险等级：P1

建议处理方向：
- 统一写为左闭右开：
  - 日：`[当天 00:00:00, 次日 00:00:00)`
  - 周：`[周一 00:00:00, 下周一 00:00:00)`
  - 月：`[当月 1 日 00:00:00, 次月 1 日 00:00:00)`
- 所有周期 ID 按 UTC+3 生成。

### 暗坑 8：每达到一个档位立即发放，但没有定义一次充值跨多档、重复达标、回滚撤销

涉及位置：
- `stay项目组.txt` L36-L39：每达到一个档位门槛，就立即发放该档位奖励。
- `stay项目组.txt` L115-L117：以金币到账时间为准，仅统计有效到账金币。

核心问题：
- 用户单笔充值可能直接跨多个档位，应发所有新达成档位，还是只发最高档？
- 同一周期内达标奖励是否每档只发一次？
- 如果充值后退款、风控撤销、金币回滚，已发奖励是否追回？
- “有效到账金币”的有效性何时最终确认？

具体场景：
- 用户月累计 0，单笔到账 30,000,000 金币。
- 月档有 600万、1000万、1500万、2000万、2500万、3000万。
- 系统应该一次发 6 个档位奖励，还是只发 3000 美金档？文档没说。

风险等级：P0

建议处理方向：
- 建议规则：同一周期内，每个档位奖励每个用户最多发一次；单次到账跨多个未领取档位时，按从低到高补发所有新达成档位。
- 建立发奖幂等键：`user_id + period_type + period_id + region + tier_id + reward_item_id`。
- 充值撤销后：未发奖励不发；已发自动道具是否追回、冻结或只做风控记录，需要产品明确。

---

### 暗坑 9：小时刷新榜单 vs 实时达标发奖，用户看到的数据可能滞后

涉及位置：
- `stay项目组.txt` L19-L23：用户停留活动页面时，榜单数据每小时自动刷新一次；刷新只更新数据，不重置浏览状态。
- `stay项目组.txt` L36-L39：达档立即发放。

核心问题：
- 档位奖励立即发，但页面数据每小时刷新。
- 用户充值后如果进度条、Top3、我的累计充值不及时更新，会出现“奖励已到账但页面没达标”或“页面达标但奖励没到账”的体验裂缝。

具体场景：
- 用户充值 100 美金后立刻回活动页。
- 发奖服务已经发 VIP，但活动页仍显示旧累计充值，需要等 1 小时刷新。
- 用户会以为没计入，联系客服。

风险等级：P1

建议处理方向：
- “每小时自动刷新”只适用于榜单全量列表。
- 我的累计充值、进度条、档位达成状态应在页面进入、充值返回、手动下拉、发奖成功通知后即时刷新。
- Top3 达标榜可以按分钟级或事件驱动刷新，至少要提供“最近更新时间”。

---

### 暗坑 10：Daily / Weekly / Monthly 三周期同时累计，奖励是否可叠加没写

涉及位置：
- `stay项目组.txt` L64-L68：Daily / Weekly / Monthly 切换后进度、档位奖励、Top3 联动刷新。
- `stay项目组.txt` L36-L39：在某个统计周期内（日/周/月）达到档位即发放。

核心问题：
- 同一笔充值天然会同时计入日、周、月三个周期。
- 如果三套档位奖励都发，成本很高；如果只发一个周期，用户又会觉得月累计没意义。
- 文档没明确三周期奖励是否可同时获得。

具体场景：
- 用户今天充值 600 美金。
- 同时达到日 300 美金档、周 600 美金档、月 600 美金档。
- 系统是否同时发日档、周档、月档三份奖励？没写。

风险等级：P0

建议处理方向：
- 明确写：同一笔有效金币到账会同时计入日/周/月累计；若分别达到日/周/月档位，则各周期档位奖励可叠加发放。
- 如果不允许叠加，必须写清楚优先级和展示逻辑。

---

### 暗坑 11：VIP、头像框、气泡、资料卡等奖励缺少资产叠加/覆盖/有效期规则

涉及位置：
- `充值档位.pdf` 视觉识别：奖励表包含 VIP、头像框、进场特效、气泡、资料卡装饰、标签、座驾、勋章、靓号等。
- `stay项目组.txt` L88-L91：奖励卡展示奖励 icon + 名称/数量。

核心问题：
- `VIP2*7`、`头像框*15`、`进场特效*10` 这类写法没有解释是等级、天数、数量，还是有效期。
- 用户已有同类资产时，是延长有效期、覆盖、升级，还是并存？
- 靓号 / Pretty ID / 定制类权益是否有库存、冲突、占用期，也没写。

具体场景：
- 用户已有 VIP5 剩余 20 天，又获得 VIP2*7。
- 是降级到 VIP2、延长 VIP5 7 天、增加 VIP2 背包道具，还是忽略？

风险等级：P1

建议处理方向：
- 奖励表字段补齐：奖励类型、奖励 ID、展示名称、数量、有效期、发放方式、叠加方式、同类资产冲突规则。
- VIP 类建议明确：若新奖励等级低于当前等级，是延长当前等级还是发放指定等级体验卡。
- 定制/靓号类必须有库存锁定和人工核销流程。

### 暗坑 12：历史榜单只写 Top3，但排行奖励覆盖 Top4-8，用户和客服无法追溯

涉及位置：
- `充值页面-榜单.txt` L18-L23：排行奖励支持查看 Top2-3、Top4-8。
- `充值页面-榜单历史.txt` L6-L12：历史只展示 Top3。

核心问题：
- 奖励实际覆盖 Top4-8，但历史榜单只展示 Top3。
- Top4-8 用户领奖后，历史页无法自证自己当期排名。
- 客服处理纠纷也缺少用户可见依据。

具体场景：
- 用户获得月榜第 6 名奖励，但历史榜单只展示 Top3。
- 用户无法从前端确认自己为什么拿到 / 没拿到 Top4-8 奖励。

风险等级：P1

建议处理方向：
- 历史页至少支持查看完整 Top8，或在“我的历史排名”展示本人当期排名与奖励发放状态。
- 客服后台必须能查周期完整 Top20 快照和发奖记录。

### 暗坑 13：系统通知只写“发放完成后发送”，没有失败、延迟、部分发放状态

涉及位置：
- `stay项目组.txt` L17-L18：奖励发放完成后，系统发送对应系统通知给用户。

核心问题：
- 发奖可能部分成功：自动奖励成功，定制权益待领取；某个道具库存不足；通知发送失败。
- 只写“发放完成后通知”无法覆盖异步任务、失败重试和客服对账。

具体场景：
- 用户达到月榜 Top1，VIP 自动发放成功，但靓号需要人工领取。
- 通知应该说“自动奖励已到账，靓号权益请联系客服”，而不是笼统“全部奖励已发放”。

风险等级：P1

建议处理方向：
- 通知模板按奖励状态拆：自动到账、待联系客服领取、发放失败处理中、资格取消。
- 发奖记录落库：待发放、发放中、已发放、部分发放、待领取、已核销、失败、已撤销。

### 暗坑 14：活动起止时间、版本生效时间没写，历史 3 周 / 3 月边界不清

涉及位置：
- 全文只定义统计周期，未定义活动开始、活动结束、活动配置生效时间。
- `充值页面-榜单历史.txt` L6-L12：历史展示近 3 周 / 3 月。

核心问题：
- 没有活动起止时间，无法判断活动上线当天是否从 00:00 补算，活动结束后是否继续发奖。
- 奖励档位如果运营修改，历史周期按旧奖励还是新奖励，也没定义。

具体场景：
- 活动 5 月 8 日 15:00 上线。
- 5 月 8 日 00:00-14:59 的充值是否计入当天 Daily？
- 5 月整月 Monthly 是否包含 5 月 1 日以来充值？文档没说。

风险等级：P1

建议处理方向：
- 增加活动配置：活动开始时间、结束时间、展示结束时间、发奖截止时间、时区。
- 统计窗口取交集：`自然周期 ∩ 活动有效期`。
- 奖励配置按周期生成快照，周期开始后不随后台修改追溯。

## 3. 建议补充的关键规则清单

1. 周期模型：`period_type（日/周/月） + period_id + UTC+3 start/end 左闭右开`。
2. 分区模型：英语区、阿语区是独立榜单还是展示筛选；奖励是否分区独立发。
3. 充值事件模型：金币到账事件、来源类型、买方/卖方账号类型、是否有效、是否撤销。
4. 档位达成模型：每周期每分区每用户每档位只记录首次达成时间。
5. 发奖幂等模型：`user_id + period + region + tier/rank + reward_item` 防重复发。
6. 排行榜快照：周期结束即冻结快照，发奖、历史、客服对账都读快照。
7. 排名排序规则：金额榜与达成榜分别定义排序和最终兜底排序。
8. 奖励资产规则：有效期、叠加、覆盖、库存、人工领取、过期、撤销。
9. 历史展示规则：至少支持 Top8 或本人历史排名与发奖状态。
10. 活动配置规则：开始/结束/展示/发奖截止时间，配置变更不追溯已冻结周期。

---

## 4. 结论

这份需求当前能表达大方向：充值奖励 + 周/月排行 + 历史榜单 + 分区 + 币商充值计入。

但还不能直接给开发开工。

最危险的不是页面，而是结算与发奖口径：

- 周期结束清空与次日发奖冲突。
- 币商/代理/金币到账口径冲突。
- 分区只写一句，没有贯穿榜单和发奖。
- 排名快照、排序兜底、奖励幂等、撤销回滚都缺。

如果这些不补，开发大概率会各自脑补，最后炸在发奖、客服投诉和运营对账上。
