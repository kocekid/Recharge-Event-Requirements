## 1. 活动目标

### 功能描述

充值活动用于鼓励用户在活动期间完成充值。

用户充值后，系统按照每日、每周、每月三个周期分别累计充值金额，并根据累计结果展示：

- 当前周期累计充值进度。
- 已达到的充值档位。
- 可获得或已获得的档位奖励。
- 达标前三名。
- 周榜和月榜排名。
- 历史榜单结果。

活动既要让用户清楚知道“还差多少可以拿奖励”，也要让客服和运营能追溯“为什么这个用户拿到了或没拿到奖励”。

### 业务流程图

用户看到活动入口
→ 点击进入充值活动页
→ 查看每日、每周、每月充值进度
→ 用户完成充值并收到金币
→ 系统判断本次金币是否计入活动
→ 计入后更新三个周期的累计充值
→ 判断是否达到新的奖励档位
→ 达到档位后处理档位奖励
→ 同步更新达标前三名
→ 同步更新周榜和月榜
→ 周期结束后形成固定榜单结果
→ 按固定榜单结果发放排行奖励
→ 结果进入历史榜单，供用户和客服查询

---

## 2. 活动范围

### 功能描述

本活动包含以下范围：

- 充值奖励页。
- 规则说明页。
- 档位奖励。
- 达标前三名展示。
- 周榜和月榜。
- 历史榜单。
- 奖励发放与领取。
- 通知提醒。
- 客服和运营查询处理。

本活动不展开后台页面细节，只保留业务上必须存在的运营处理能力。

### 业务流程图

活动配置完成
→ 用户进入活动页面
→ 用户充值
→ 系统计入有效充值
→ 用户获得档位奖励或排行资格
→ 系统发放奖励或生成待领取资格
→ 用户查看奖励、榜单和历史结果
→ 客服或运营可查询并处理异常

## 3. 活动时间规则

### 功能描述

活动需要明确开始时间、结束时间和展示结束时间。

活动未开始时，用户不能参与累计。

活动进行中时，用户充值后可参与累计、达档和排名。

活动结束后，不再新增充值累计，但用户仍可查看历史榜单和处理未完成奖励。

展示结束后，活动归档，不再作为当前活动展示。

每日、每周、每月三个周期分别计算：

- 每日：当天零点开始，到次日零点前结束。
- 每周：每周一零点开始，到下周一零点前结束。
- 每月：每月一日零点开始，到下月一日零点前结束。

如果活动开始或结束时间落在周期中间，实际统计范围以活动有效时间和自然周期重叠的部分为准。

### 业务流程图

活动未开始
→ 到达活动开始时间
→ 活动进行中
→ 用户充值可参与累计
→ 到达活动结束时间
→ 停止新增充值累计
→ 用户继续查看历史榜单和未完成奖励
→ 到达展示结束时间
→ 活动归档

---

## 4. 分区规则

### 功能描述

活动区分英语区和阿语区。

用户进入活动页时，只能看到自己所在分区对应的榜单、排名、达标前三名和历史榜单。

分区会影响以下内容：

- 当前活动展示。
- 周榜和月榜排名。
- 达标前三名。
- 排行奖励发放范围。
- 历史榜单查询范围。

如果用户活动期间分区发生变化，本期需要产品最终确认采用哪种规则：

- 按充值发生时的分区计算。
- 按用户当前分区计算。
- 活动期间锁定首次参与时的分区。

为了避免榜单和奖励争议，建议采用“充值发生时的分区计算”。

### 业务流程图

用户进入活动页
→ 系统识别用户当前分区
→ 展示对应分区活动内容
→ 用户充值
→ 系统按确认后的分区规则计入对应分区
→ 更新该分区奖励进度和榜单
→ 周期结束后按该分区结果发放奖励
→ 历史榜单按分区保存和查询

---

## 5. 充值统计规则

### 功能描述

活动统计以用户实际收到金币为准。

普通用户通过官方充值或向币商购买金币，只要金币成功到账，且没有被风控判定为无效，就可以计入活动。

以下情况不应计入活动：

- 充值失败。
- 金币未到账。
- 退款或撤销。
- 风控判定无效。
- 产品明确排除的账号或充值方式。

代理或币商账号自身交易是否计入，需要产品明确。不能默认把所有交易都计入，否则可能出现刷榜或奖励争议。

### 业务流程图

金币到账
→ 判断用户是否有活动资格
→ 判断金币来源是否允许计入
→ 判断是否存在退款、撤销或风控限制
→ 符合规则
→ 计入用户每日、每周、每月累计充值
→ 刷新奖励进度
→ 刷新达标记录
→ 刷新榜单排名

### 退款与异常流程图

已计入活动的充值发生退款或撤销
→ 系统重新计算用户累计充值
→ 判断用户是否仍达到已获得奖励的条件
→ 如果仍达到条件，奖励保持不变
→ 如果不再达到条件，标记需要运营处理
→ 如果奖励尚未发放，停止发放
→ 如果奖励已经发放，进入人工核查或追回流程
→ 记录处理结果，供客服解释

## 6. 活动入口与页面导航

### 功能描述

用户从活动入口进入后，默认进入充值奖励页。

页面顶部需要提供以下入口：

- 充值奖励。
- 规则说明。
- 充值排行榜。
- 历史榜单。

用户可以在不同页面之间切换。切换页面时，应保留当前活动和分区范围，不应跳到错误活动或错误分区。

### 业务流程图

用户点击活动入口
→ 进入充值奖励页
→ 查看当前周期充值进度
→ 点击规则说明
→ 查看活动规则
→ 点击充值排行榜
→ 查看周榜或月榜
→ 点击历史榜单
→ 查看历史周期结果
→ 返回充值奖励页
→ 继续查看进度或充值

## 7. 充值奖励页

### 功能描述

充值奖励页是活动主页面。

页面需要展示：

- 每日、每周、每月三个周期入口。
- 当前周期剩余时间。
- 我的累计充值金额。
- 我的奖励进度。
- 各档位奖励内容。
- 已达成、未达成、已领取、待领取等奖励处理阶段。
- 达标前三名入口或展示。

用户切换每日、每周、每月时，页面内容应同步切换为对应周期的数据。

用户充值后，页面应尽快刷新累计金额和进度，避免用户误以为充值没有生效。

### 业务流程图

用户进入充值奖励页
→ 默认展示当前周期
→ 用户查看累计充值和奖励进度
→ 用户切换每日、每周、每月
→ 页面展示对应周期内容
→ 用户完成充值
→ 系统计入有效充值
→ 页面刷新累计充值
→ 判断是否达到新档位
→ 达到后展示奖励结果
→ 用户查看或领取对应奖励

### 加载失败流程图

用户进入页面
→ 页面内容加载失败
→ 展示重试提示
→ 用户点击重试
→ 再次加载活动内容
→ 加载成功后展示页面
→ 多次失败时提示稍后再试或联系客服

---

## 8. 档位奖励规则

### 功能描述

档位奖励按每日、每周、每月分别配置。

用户在某个周期内累计充值达到档位要求后，可以获得该档位奖励。

同一周期内，同一档位只能获得一次。

如果用户一次充值跨过多个未达成档位，应从低到高补齐所有新达成档位，而不是只发最高档。

奖励分两类：

- 可直接发放的奖励：例如金币、普通道具、固定权益。
- 需要人工处理的奖励：例如靓号、定制权益、需要运营确认的特殊奖励。

需要人工处理的奖励，不应直接自动发放，而是生成待领取资格，由运营或客服继续处理。

### 业务流程图

用户金币到账
→ 更新每日、每周、每月累计充值
→ 判断是否达到新的奖励档位
→ 没有达到新档位
→ 只刷新页面进度

用户金币到账
→ 达到一个或多个新档位
→ 从低到高处理所有新达成档位
→ 判断奖励是否可以直接发放
→ 可以直接发放的奖励立即发放
→ 需要人工处理的奖励生成待领取资格
→ 通知用户奖励处理结果
→ 页面展示最新奖励阶段

### 跨档示例流程图

用户原累计充值未达到第一档
→ 用户一次充值后同时达到第一档、第二档、第三档
→ 系统依次处理第一档、第二档、第三档
→ 每个档位只处理一次
→ 已处理档位后续不再重复处理

---

## 9. 达标前三名

### 功能描述

达标前三名用于展示每个周期内最早达到指定目标的前三位用户。

每日、每周、每月可以有不同的达标目标。

达标前三名不是普通充值金额排行榜，它看的是“谁最早达到目标”。

排序建议：

- 先按达标时间，越早越靠前。
- 如果达标时间相同，再按累计充值金额，越高越靠前。
- 如果仍然相同，再按用户财富等级或账号编号作为兜底顺序。

### 业务流程图

用户充值到账
→ 系统更新累计充值
→ 判断是否首次达到指定目标
→ 未达到目标
→ 不进入达标前三名候选

用户充值到账
→ 首次达到指定目标
→ 记录首次达标时间
→ 与当前已达标用户排序
→ 生成前三名展示结果
→ 页面展示前三名用户和奖励说明

---

## 10. 充值排行榜

### 功能描述

充值排行榜展示用户在周周期和月周期内的累计有效充值排名。

排行榜需要按分区展示，用户只看自己分区的榜单。

页面需要展示：

- 当前榜单周期。
- 剩余时间。
- 用户头像、昵称、排名和累计充值。
- 我的当前排名和累计充值。
- 排行奖励入口或说明。

排行榜应与奖励发放范围保持一致。如果奖励发到第四至第八名，页面或历史记录也要能追溯这些名次，否则客服无法解释奖励来源。

### 业务流程图

用户进入排行榜页
→ 系统识别用户分区
→ 用户选择周榜或月榜
→ 展示对应周期榜单
→ 用户充值后累计变化
→ 系统重新计算排名
→ 页面刷新榜单和我的排名
→ 周期结束
→ 形成固定榜单结果
→ 后续发奖和历史查询都按该结果处理

### 榜单排序流程图

统计用户周期内有效充值
→ 按累计充值金额从高到低排序
→ 金额相同时按达到该金额的时间排序
→ 仍相同时按财富等级或账号编号排序
→ 得到榜单顺序
→ 展示给用户

---

## 11. 排行奖励规则

### 功能描述

排行奖励根据周期结束后的固定榜单结果发放。

周期未结束前，榜单排名只是当前展示，不应作为最终发奖依据。

周期结束后，系统需要形成固定榜单结果。之后即使页面切到新周期，也不能删除旧周期结果。

排行奖励、历史榜单、客服查询和用户通知，都应使用同一份固定结果，避免出现页面看到一套、发奖按另一套的情况。

如果同一用户同时满足周榜和月榜奖励，是否都可获得，需要产品明确。若允许叠加，应在规则页清楚说明。

### 业务流程图

周周期或月周期结束
→ 系统停止该周期新增累计
→ 形成固定榜单结果
→ 判断每个名次对应奖励
→ 判断奖励是否可直接发放
→ 可直接发放的奖励立即发放
→ 需要人工处理的奖励生成待领取资格
→ 通知用户奖励结果
→ 将结果写入历史榜单
→ 客服可按固定结果查询和解释

### 异常处理流程图

周期结束后发现用户充值异常
→ 运营或风控发起核查
→ 判断是否影响榜单结果
→ 如果不影响，维持原结果
→ 如果影响，按人工处理规则修正
→ 修正后记录原因和处理人
→ 必要时通知受影响用户

---

## 12. 历史榜单

### 功能描述

历史榜单用于查看已经结束的周榜和月榜结果。

历史榜单应按分区保存。

历史榜单需要展示：

- 榜单周期。
- 分区。
- 用户排名。
- 用户累计充值。
- 奖励资格或奖励结果。

历史榜单不应随着用户后续退款、换区、昵称修改或新周期充值而被随意改变。确需修正时，应由运营按人工处理流程修正，并保留处理说明。

### 业务流程图

用户进入历史榜单页
→ 选择周榜或月榜
→ 选择历史周期
→ 系统读取该周期固定榜单结果
→ 展示历史排名和奖励结果
→ 用户查看自己的历史排名
→ 如有疑问，联系客服查询同一份固定结果

---

## 13. 规则说明页

### 功能描述

规则说明页用于向用户解释活动玩法。

规则页需要用用户能理解的语言说明：

- 活动时间。
- 每日、每周、每月如何计算。
- 哪些充值会计入活动。
- 哪些充值不会计入活动。
- 档位奖励如何获得。
- 排行奖励如何获得。
- 奖励什么时候发放。
- 特殊奖励如何领取。
- 异常、退款、风控情况下如何处理。

规则页文案必须和实际处理规则一致，不能为了短而省略关键限制。

### 业务流程图

用户进入规则页
→ 查看活动时间
→ 查看充值计入规则
→ 查看档位奖励规则
→ 查看排行榜规则
→ 查看奖励发放说明
→ 用户理解规则后返回活动页
→ 用户按规则参与充值活动

---

## 14. 奖励处理阶段

### 功能描述

奖励从产生到完成，需要明确处理阶段。

建议使用以下业务阶段：

- 未获得：用户尚未达到条件。
- 已获得待处理：用户达到条件，但奖励还未完成处理。
- 已发放：奖励已经到账。
- 待领取：用户已获得资格，但需要人工处理或用户补充信息。
- 已领取：用户已完成领取。
- 发放失败：奖励处理失败，需要重试或人工处理。
- 已取消：因退款、风控或人工确认，奖励不再发放。

页面要让用户看懂当前奖励处于哪个阶段，以及下一步该做什么。

### 业务流程图

用户达到奖励条件
→ 系统判断奖励类型
→ 普通奖励进入待处理
→ 系统完成发放
→ 页面展示已发放

用户达到奖励条件
→ 特殊奖励进入待领取
→ 用户或客服完成领取信息确认
→ 运营处理奖励
→ 页面展示已领取

奖励处理失败
→ 系统提示失败原因
→ 可重试则重新处理
→ 不可重试则进入人工处理
→ 人工处理完成后更新结果

---

## 15. 通知规则

### 功能描述

当用户获得重要结果时，系统应通知用户。

建议通知场景：

- 达到新的充值档位。
- 奖励已发放。
- 获得待领取特殊奖励。
- 周榜或月榜获得排行奖励。
- 奖励处理失败，需要联系客服。
- 活动即将结束。

通知内容要明确告诉用户：获得了什么、来自哪个周期、下一步是否需要操作。

### 业务流程图

用户触发奖励或榜单结果
→ 系统判断是否需要通知
→ 生成通知内容
→ 发送给用户
→ 用户点击通知
→ 进入对应活动页或奖励说明页
→ 用户查看结果或继续领取

---

## 16. 页面刷新规则

### 功能描述

用户充值后，页面应尽快刷新充值累计、奖励进度和排名。

如果榜单不是实时刷新，也要保证“我的累计充值”和“我的奖励进度”尽快更新。

否则用户会认为充值没有计入活动，容易造成客服投诉。

周期结束时，当前周期页面应切换到新周期，同时保留旧周期结果进入历史榜单。

### 业务流程图

用户停留在活动页
→ 用户完成充值
→ 系统收到金币到账结果
→ 页面刷新我的累计充值
→ 页面刷新奖励进度
→ 页面刷新我的排名
→ 如果榜单延迟刷新，页面给出合理提示
→ 周期结束后页面切到新周期
→ 旧周期结果进入历史榜单

## 17. 客服与运营处理能力

### 功能描述

为了支撑活动落地，客服和运营需要能查询和处理关键问题。

至少需要支持：

- 查询用户某周期累计充值。
- 查询用户充值是否计入活动。
- 查询用户奖励获得和发放结果。
- 查询用户榜单排名。
- 查询历史榜单固定结果。
- 处理特殊奖励领取。
- 处理退款、撤销、风控导致的奖励争议。
- 人工修正异常结果并记录原因。

本版不展开后台页面，只说明这些业务能力必须存在。

### 客服处理流程图

用户咨询奖励问题
→ 客服查询用户和活动周期
→ 查看用户有效充值记录
→ 查看用户奖励达成情况
→ 查看奖励处理阶段
→ 如果规则正常，向用户解释原因
→ 如果存在异常，提交运营处理
→ 运营处理后更新结果
→ 客服回复用户

### 风控处理流程图

系统或人工发现异常充值
→ 标记充值需要核查
→ 判断是否已计入活动
→ 判断是否已发放奖励
→ 未发放奖励则暂停处理
→ 已发放奖励则进入人工核查
→ 核查确认有效则恢复奖励
→ 核查确认无效则取消或追回奖励
→ 记录处理原因

## 18. 待产品确认项

### 功能描述

以下问题需要产品最终确认，否则开发和运营执行时容易出现分歧：

1. 用户活动期间换区时，充值到底按哪个分区计入。
2. 代理或币商账号自身交易是否计入活动。
3. 普通用户向币商购买金币是否全部计入活动。
4. 周榜和月榜奖励是否可以同时获得。
5. 单笔充值跨多个档位时，是否补发所有新达成档位。本版建议补发全部新达成档位。
6. 特殊奖励由谁处理、处理时限多久、用户需要提供什么信息。
7. 退款或风控发生在奖励发放之后，奖励是追回、抵扣，还是只做人工记录。
8. 历史榜单展示名次是否必须覆盖全部奖励名次。
9. 榜单金额相同、达成时间相同时，最终排序采用什么兜底规则。
10. 活动结束后，未领取特殊奖励保留多久。

### 业务流程图

发现规则未明确
→ 产品确认最终规则
→ 同步更新活动说明
→ 同步更新客服解释口径
→ 同步更新运营处理方式
→ 活动上线前完成核对
