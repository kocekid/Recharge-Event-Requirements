# 充值活动发送奖励通知消息模板（详细版）

文档版本：V1.0  
生成目的：补充充值活动奖励发放系统消息模板详细规则  
适用范围：日充值奖励、周充值奖励、月充值奖励、周排行奖励、月排行奖励  
关联主文档：`/Users/mac/心音跳动/充值活动/充值活动PRD_重写版.md`

---

## 1. 背景说明

当前主 PRD 中的“奖励发放系统消息”文案过于简化，只能表达“你获奖了”，但无法让用户在消息内直接看清：

- 获得了什么奖励
- 每个奖励的数量 / 天数是多少
- 哪些奖励已自动到账
- 哪些奖励需要人工处理或联系客服领取
- 多语言场景下标题与奖励项如何动态翻译

这会直接导致：

- 用户收到消息后仍要回活动页二次确认
- 客服需要反复解释“到底给了什么”
- 靓号、Banner 等人工类权益缺少明确提示
- 多奖励组合场景文案无法复用

所以这里单独补一份“发送奖励通知消息模板详细版”，用于指导产品、服务端、客户端、多语言配置和测试联调。

---

## 2. 目标

系统消息必须支持把一次奖励发放中的全部奖励项完整列出来。标题中的变量也不能写死，但这里的变量不是“金币 / VIP / 装扮”这种奖励明细项，而是本次奖励所属的奖励项名称，例如“本月达标奖励”“上周第{rank}名冲榜奖励”“上月第{rank}名冲榜奖励”“本周达标奖励”。

目标效果示例：

```text
标题：恭喜获得充值活动本月达标奖励
奖励：
金币*100000
炫彩头像框*7天
VIP*7天
靓号*30天
周榜Banner奖励*7天
```

注意：
- 上面只是展示结构，不代表每次都包含全部奖励项。
- 标题里的变量应表示本次奖励所属的奖励项名称，例如“本月达标奖励”“上周第{rank}名冲榜奖励”，不能拿奖励明细项名称去拼标题。
- 每次消息只展示本次发放/解锁的奖励项，不展示未获得项。

## 3. 消息结构要求

### 3.1 基础结构

每条奖励通知消息由以下两部分组成：

1. 标题
2. 奖励明细正文

标准结构：

```text
标题：恭喜获得充值活动{reward_title_name}
奖励：
{reward_line_1}
{reward_line_2}
{reward_line_3}
...
```

### 3.2 标题规则

标题格式固定为：

```text
恭喜获得充值活动{reward_title_name}
```

其中：
- `{reward_title_name}` 为本次奖励所属的奖励项名称，不写死。
- 该变量来源于奖励来源类型或奖励场景配置，并参考现有系统消息表述做统一收口。
- 标题要尽量与现有正文语义一致，避免正文说“昨日充值达到奖励等级”，标题却写成另一个风格，搞得像两套产品写的。
- 标题不按金币、VIP、装扮、靓号等明细项进行摘要拼接。
- 标题不直接照搬整句正文，而是提炼出“充值活动 + 奖励归属名 + 奖励结果”的短标题。

#### 标题优化原则

基于你现有的系统消息正文：

- 恭喜你昨日充值达到奖励等级，其他奖励已自动发放。
- 恭喜你本周充值达到奖励等级，奖励已发放，请查收。
- 恭喜你本月充值达到奖励等级，其他奖励已自动发放，若有靓号奖励请联系管理领取。
- 恭喜您在充值活动获得上周第 # 名！奖励已发放，请查收。
- 恭喜您在充值活动获得上月第 # 名！奖励已发放，请查收。

标题建议不要继续用“日充值达标奖励 / 周榜奖励”这种太后台配置味的名字，而应改成更贴近正文语义的标题名。

#### 标题生成示例

| 奖励场景 | 现有正文语义 | 优化后标题示例 |
|---|---|---|
| 日充值档位发奖 | 昨日充值达到奖励等级 | 恭喜获得充值活动昨日达标奖励 |
| 周充值档位发奖 | 本周充值达到奖励等级 | 恭喜获得充值活动本周达标奖励 |
| 月充值档位发奖 | 本月充值达到奖励等级 | 恭喜获得充值活动本月达标奖励 |
| 周榜排行发奖 | 获得上周第 # 名 | 恭喜获得充值活动上周第{rank}名冲榜奖励 |
| 月榜排行发奖 | 获得上月第 # 名 | 恭喜获得充值活动上月第{rank}名冲榜奖励 |

### 3.3 正文规则

正文第一行固定文案：

```text
奖励：
```

其后逐行输出奖励项明细。每个奖励项单独占一行，格式如下：

```text
{reward_display_name}*{reward_display_value}
```

示例：

```text
奖励：
金币*100000
炫彩头像框*7天
VIP*7天
靓号*30天
周榜Banner奖励*7天
```

## 4. 奖励项明细格式规则

### 4.1 通用规则

每个奖励项必须输出两个信息：

1. 奖励展示名
2. 展示数量 / 时长

统一格式：

```text
{奖励展示名}*{数值}
```

禁止写法：
- 金币奖励100000
- 赠送VIP7天
- 恭喜获得靓号，时长30天

原因：格式不统一，客户端渲染、测试校验、多语言映射都会变得很恶心。统一成 `名称*值`，省得后面一地鸡毛。

### 4.2 奖励类型映射规则

| 奖励类型 | 展示名规则 | 展示值规则 | 示例 |
|---|---|---|---|
| 金币 | 固定显示“金币” | 显示金币数量，不带天数 | 金币*100000 |
| 装扮 | 显示具体装扮名称 | 显示`x天` | 炫彩头像框*7天 |
| VIP | 固定显示“VIP”或具体 VIP 名称 | 显示`x天` | VIP*30天 |
| 靓号 | 固定显示“靓号”或具体靓号名称 | 显示`x天` | 靓号*30天 |
| 周榜 Banner 奖励 | 固定显示“周榜Banner奖励” | 显示`x天` | 周榜Banner奖励*7天 |
| 月榜 Banner 奖励 | 固定显示“月榜Banner奖励” | 显示`x天` | 月榜Banner奖励*30天 |

### 4.3 金币奖励格式

金币奖励只展示数量，不展示“天”。

格式：

```text
金币*{coin_amount}
```

示例：

```text
金币*50000
金币*100000
金币*300000
```

### 4.4 装扮奖励格式

装扮奖励必须展示具体装扮名称，不能只写“装扮”。

格式：

```text
{装扮名称}*{days}天
```

示例：

```text
星耀座驾*7天
霓虹头像框*15天
王者气泡框*30天
```

### 4.5 VIP 奖励格式

格式：

```text
VIP*{days}天
```

如果产品后续区分 VIP 等级，也可扩展为：

```text
{vip_name}*{days}天
```

示例：

```text
VIP*7天
VIP*30天
至尊VIP*30天
```

### 4.6 靓号奖励格式

格式：

```text
靓号*{days}天
```

如果运营希望展示具体靓号资产名，也可扩展为：

```text
{pretty_id_name}*{days}天
```

示例：

```text
靓号*7天
靓号*30天
8888靓号*30天
```

### 4.7 Banner 奖励格式

Banner 奖励必须区分周榜和月榜，不允许都写成一个泛词“Banner奖励”，不然客服后面对账又要骂娘。

格式：

```text
周榜Banner奖励*{days}天
月榜Banner奖励*{days}天
```

示例：

```text
周榜Banner奖励*7天
月榜Banner奖励*30天
```

## 5. 多奖励组合规则

### 5.1 组合展示原则

一次奖励发放可能同时包含多个奖励项，例如：

- 金币 + VIP
- 金币 + 装扮 + VIP
- 金币 + 靓号 + Banner
- 装扮 + VIP + 靓号 + Banner

此时消息正文必须完整列出全部奖励项，每个奖励项一行，不允许合并成一行长句。

正确示例：

```text
标题：恭喜获得充值活动本月达标奖励
奖励：
金币*100000
炫彩头像框*7天
VIP*7天
靓号*30天
周榜Banner奖励*7天
```

错误示例：

```text
恭喜你获得了100000金币、7天VIP、7天头像框、30天靓号和7天Banner奖励。
```

这玩意儿读起来像运营群里临时手打一坨，别拿去做系统模板。

### 5.2 奖励顺序规则

为了保证用户理解顺序一致，奖励项展示顺序固定如下：

1. 金币
2. 装扮
3. VIP
4. 靓号
5. 周榜 Banner 奖励
6. 月榜 Banner 奖励
7. 其他后续扩展奖励

原因：
- 金币最直观，优先展示。
- 装扮 / VIP 属于自动到账类，紧随其后。
- 靓号 / Banner 往往涉及人工处理或运营权益，放在后面更容易单独提示。

### 5.3 同类奖励合并规则

| 场景 | 规则 |
|---|---|
| 同一奖励类型出现 1 条 | 直接输出 1 行 |
| 同一奖励类型出现多条且名称相同、时长相同 | 服务端可在发消息前合并为 1 行 |
| 同一奖励类型出现多条但名称不同 | 必须分多行展示 |
| 同一奖励类型出现多条但时长不同 | 必须分多行展示 |

示例：

```text
奖励：
金币*100000
霓虹头像框*7天
王者气泡框*15天
VIP*7天
```

## 6. 多语言支持要求

### 6.1 多语言目标

该消息模板必须支持多语言，至少满足：

- 简体中文
- 英文
- 阿拉伯语

消息模板键保持稳定，具体文案通过多语言资源表配置，不允许为每种语言创建不同模板键。别把一个简单模板搞成三套分叉屎山。

### 6.2 多语言拆分方式

建议将消息拆成以下可翻译单元：

| 资源键 | 含义 | 中文示例 | 英文示例 |
|---|---|---|---|
| reward_notice_title | 标题模板 | 恭喜获得充值活动{reward_title_name} | Congratulations! You received {reward_title_name} |
| reward_notice_label | 正文标签 | 奖励： | Rewards: |
| reward_name_coin | 金币名称 | 金币 | Coins |
| reward_name_vip | VIP名称 | VIP | VIP |
| reward_name_pretty_id | 靓号名称 | 靓号 | Pretty ID |
| reward_name_weekly_banner | 周榜Banner奖励名称 | 周榜Banner奖励 | Weekly Banner Reward |
| reward_name_monthly_banner | 月榜Banner奖励名称 | 月榜Banner奖励 | Monthly Banner Reward |
| reward_title_daily_tier | 昨日达标奖励标题名 | 昨日达标奖励 | Yesterday Milestone Reward |
| reward_title_weekly_tier | 本周达标奖励标题名 | 本周达标奖励 | Weekly Milestone Reward |
| reward_title_monthly_tier | 本月达标奖励标题名 | 本月达标奖励 | Monthly Milestone Reward |
| reward_title_weekly_ranking | 上周第{rank}名冲榜奖励标题名 | 上周第{rank}名冲榜奖励 | Last Week Top {rank} Ranking Reward |
| reward_title_monthly_ranking | 上月第{rank}名冲榜奖励标题名 | 上月第{rank}名冲榜奖励 | Last Month Top {rank} Ranking Reward |
| reward_duration_day | 天单位 | 天 | day(s) |

### 6.3 多语言变量规则

| 变量 | 说明 |
|---|---|
| `{reward_title_name}` | 标题中的奖励项名称，例如昨日达标奖励、本周达标奖励、上周第{rank}名冲榜奖励 |
| `{reward_display_name}` | 奖励项名称 |
| `{reward_display_value}` | 奖励数量或时长 |
| `{days}` | 时长数值 |
| `{coin_amount}` | 金币数量 |

### 6.4 英文示例

```text
Title: Congratulations! You received Recharge Activity Monthly Milestone Reward
Rewards:
Coins*100000
Neon Avatar Frame*7 days
VIP*7 days
Pretty ID*30 days
Weekly Banner Reward*7 days
```

### 6.5 阿语支持要求

阿语属于 RTL（从右往左）语言，客户端渲染时需保证：

- 标题和正文支持 RTL 排版
- `*` 分隔符在阿语环境下不应被错误打散
- 数字与单位拼接后仍可正确显示
- 奖励明细建议按“名称 + `*` + 数值”整体作为不可拆分片段处理

### 6.6 多语言禁忌

禁止下面这些骚操作：

1. 中文模板写死，英文/阿语靠机器翻译即时转
2. 不同语言使用不同消息模板键
3. 标题里的奖励摘要不翻译，只翻译正文
4. 把“7天”这种值整体写死，导致别的语言无法拆单位

## 7. 变量与数据结构要求

### 7.1 单条奖励项建议字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| reward_type | string | 是 | 奖励类型：coin / decoration / vip / pretty_id / weekly_banner / monthly_banner |
| reward_name | string | 是 | 奖励名称。金币可为空后端映射，装扮必须传具体名称 |
| reward_amount | integer | 否 | 金币数量 |
| reward_days | integer | 否 | 时长天数 |
| reward_sort | integer | 是 | 展示顺序 |
| grant_mode | string | 否 | auto / service / operation |
| grant_status | string | 是 | success / partial_success / waiting_claim / failed |

```json
{
  "message_template_key": "recharge_activity_reward_detail",
  "language": "zh-CN",
  "title": "恭喜获得充值活动本月达标奖励",
  "reward_title_name": "本月达标奖励",
  "reward_lines": [
    "金币*100000",
    "炫彩头像框*7天",
    "VIP*7天",
    "靓号*30天",
    "周榜Banner奖励*7天"
  ]
}
```

### 7.3 标题名称生成规则

| 场景 | 生成规则 |
|---|---|
| 日充值档位奖励 | 固定取“昨日达标奖励” |
| 周充值档位奖励 | 固定取“本周达标奖励” |
| 月充值档位奖励 | 固定取“本月达标奖励” |
| 周排行奖励 | 固定取“上周第{rank}名冲榜奖励” |
| 月排行奖励 | 固定取“上月第{rank}名冲榜奖励” |
| 后续新增奖励来源 | 由配置表新增对应标题名称，不复用明细奖励项名称 |

建议：
- 标题名称由奖励来源类型统一控制，并尽量贴近正文已有表述。
- 正文再展示具体明细奖励项。
- 不允许把金币、VIP、装扮、靓号等明细项名称直接拿去当标题变量。
- 榜单奖励标题必须带上具体名次，直接从排行结果变量中取值，不要写成泛化的“榜单奖励”。

示例：

| 奖励来源 | 奖励明细 | 标题名称 |
|---|---|---|
| 月充值档位奖励 | 金币*100000 + VIP*7天 | 本月达标奖励 |
| 周排行奖励（第1名） | 金币*300000 + 周榜Banner奖励*7天 | 上周第1名冲榜奖励 |
| 月排行奖励（第3名） | 金币*500000 + 月榜Banner奖励*30天 + 靓号*30天 | 上月第3名冲榜奖励 |

## 8. 发放状态与文案关系

### 8.1 成功发放

当奖励发放状态为 `success` 时：
- 正常发送完整奖励明细消息。
- 自动到账类奖励直接展示结果。

### 8.2 部分成功发放

当奖励发放状态为 `partial_success` 时：
- 已自动到账的奖励项照常展示。
- 对人工处理类奖励仍可展示对应奖励项。
- 如主 PRD 已约定“人工权益联系客服领取”，则可在消息尾部追加说明。

追加说明示例：

```text
请联系客服领取需人工处理的奖励。
```

### 8.3 失败发放

当奖励发放状态为 `failed` 时：
- 不发送“恭喜获得”模板。
- 进入补偿队列或人工处理流程。

别他妈奖励没发出去还给用户发“恭喜”，那不是通知，是诈骗。

这些键可以继续保留，用于区分奖励来源；但具体消息内容建议统一走“详细奖励模板渲染逻辑”，而不是每种奖励来源手写一句固定话术。

建议改造方式：

| 现状 | 问题 | 建议 |
|---|---|---|
| 每个模板键对应一句固定话术 | 无法列出详细奖励项 | 改为统一详细模板 + 动态奖励行 |
| 排行奖励只插入排名 | 用户看不到奖励内容 | 排名信息可作为附加变量，奖励明细单独输出 |
| 多语言只写“翻译文档为准” | 缺少字段级规则 | 补充资源键、变量规则、RTL 要求 |

---

## 11. 验收标准

| 验收项 | 验收要求 |
|---|---|
| 标题动态化 | 标题中的奖励项名称不可写死，必须来自奖励来源配置，例如本月达标奖励、上周第{rank}名冲榜奖励 |
| 奖励明细完整性 | 本次发放的每个奖励项都必须逐行展示 |
| 展示格式统一 | 所有奖励项必须使用 `名称*值` 结构 |
| 金币格式正确 | 金币只显示数量，不带天数 |
| 时长格式正确 | 装扮/VIP/靓号/Banner 必须显示 `x天` |
| Banner 区分正确 | 周榜 Banner 与月榜 Banner 必须文案区分 |
| 多语言可配置 | 标题、正文标签、奖励名称、单位、后缀均可配置 |
| 模板键稳定 | 不因语言不同而变更消息模板键 |
| RTL 可显示 | 阿语场景下标题与奖励明细不乱序 |
| 部分成功可扩展 | 可支持自动奖励已发放、人工权益待领取的扩展说明 |

---

## 12. 最终结论

这版详细模板的核心不是“把文案写长一点”。

核心是把奖励通知从一句糊弄鬼的话，升级成：

- 可列出全部奖励项
- 可支持多语言
- 可支持多奖励组合
- 可兼容自动到账和人工权益
- 可直接指导服务端模板渲染与客户端展示

不然用户收到一句“奖励已发放，请查收”，查收个屁。查空气吗。
